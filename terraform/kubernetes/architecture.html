dd-observability-pov — kubernetes Architecture

Java + Python を EKS on EC2 上の Kubernetes Pod で実行する構成。Datadog Operator による SSI（Single Step Instrumentation）を Admission Controller 経由で自動注入。
📦 Region: ap-northeast-1 🏷️ Prefix: pov (variable: name_prefix) ☸️ EKS: 1.33 / Managed Node Group (Standard Support) 🌐 NAT Gateway: あり（EKS ノードの Outbound 用）

Architecture Diagram

kubectl port-forward で java-app に接続。Java → Python → PostgreSQL の 4 階層トレースを Kubernetes Pod 上で再現。 Datadog Operator の Admission Controller が APM ライブラリを init container 経由で自動注入するためコード変更は不要。

Datadog Integration — Operator + SSI

Before / After POV ストーリー

Phase状態操作
Before java-app / python-app が EKS 上で稼働。Datadog Agent 未インストール。可観測性ゼロ。 ECR へビルド・プッシュ → kubectl apply でデプロイ
After Datadog Operator の Admission Controller が Pod 再起動時に APM ライブラリを init container として自動注入。アプリのコード変更・Dockerfile 変更は一切不要。 kubectl create secret generic datadog-secret → kubectl apply -f manifests/datadog-agent.yaml → kubectl rollout restart

SSI の仕組み（Kubernetes）

  • Datadog Operator が DatadogAgent CRD を監視し、DaemonSet・Admission Controller を自動構成
  • Admission Controller (MutatingWebhook) が apm.instrumentation.enabledNamespaces: [pov] で指定した Namespace の Pod 作成を検知
  • Pod spec に init container（dd-lib-java-init / dd-lib-python-init）が自動挿入され、Agent バイナリを共有ボリューム経由で起動時に注入
  • DD_LOGS_INJECTION=true 環境変数と ad.datadoghq.com/<container>.logs アノテーションで ログ ↔ トレース相関を実現
  • mutateUnlabelled: false で DD ラベルのない Pod は変更しない（安全側）

期待されるトレース（After フェーズ）

  • Service Map: pov-java-front → pov-py-back → postgres の 3 階層（Pod が別れていても同一トレースに統合）
  • /db/normal: 1 トレース内に postgres.query span が 1 個
  • /db/n1: 1 トレース内に postgres.query span が 21 個（N+1 アンチパターン）
  • /db/long-run: postgres.query span が 35 秒幅で表示
  • Logs: JSON stdout に dd.trace_id / dd.span_id が埋まり、Datadog UI でトレース ↔ ログをクリックで行き来できる
  • Kubernetes: OrchestratorExplorer + KubeStateMetricsCore で Pod / Deployment / Node メトリクスを収集

log/trace 相関の設定

Java (LogstashEncoder)

logback-spring.xml:
<encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
# MDC から dd.trace_id / dd.span_id を自動出力
# DD_LOGS_INJECTION=true で有効化

Python (JSON formatter)

app.py:
class JsonFormatter(logging.Formatter):
    def format(self, record):
        entry = {"dd.trace_id": record.__dict__.get("dd.trace_id", "0"), ...}
        return json.dumps(entry)
# requirements.txt に ddtrace を含めない（SSI が init container 注入）
# DD_LOGS_INJECTION=true で有効化

注意: Python requirements.txt

Python アプリの requirements.txt に ddtrace を記載しないこと。 SSI が init container 経由で ddtrace を注入するため、requirements.txt に含めるとバージョン競合で ddtrace が無効化され、dd.trace_id=0（相関不可）になる。

Operations

Apply

cd terraform/kubernetes
terraform init
terraform plan
terraform apply     # EKS クラスター作成に 15〜20 分 / RDS に 10〜15 分
                    # Datadog Operator は helm_release で自動インストール

Destroy（完全クリーンアップ）

# Step 1: アプリ namespace を先に削除（推奨）
kubectl delete namespace <name_prefix> --ignore-not-found

# Step 2: Terraform で全 AWS リソースを削除
terraform destroy

自動クリーンアップされるもの:
・helm_release → Terraform が helm uninstall datadog-operator を自動実行
・ECR リポジトリ → force_delete=true でイメージが残っていても削除可
・RDS → skip_final_snapshot / deletion_protection=false で即時削除
・EKS / VPC / NAT GW / Subnet → 依存関係順に自動削除（約 15〜20 分）

コピペ手順は terraform output demo_destroy で確認できます。
