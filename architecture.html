Analytics endpoints (Java → Python → Postgres) — エラー分析シナリオ

order_monthly_stats テーブルを使った前月比成長率計算。user_id % 7 == 0 の 7 名は先月データなし → ZeroDivisionError → ~14% エラー。Java 側は /analytics のみ 500ms 待機してリトライ。

Public PathPython PathBehaviorResult

Datadog Integration (SSI)

Before / After POV ストーリー

Phase状態操作
Before Java/Spring Boot + Python が連携動作。Datadog Agent 未インストール。可観測性ゼロ。 —
After SSI で Agent インストール後、アプリ再起動するだけで APM・ログ・メトリクスを取得 DD_APM_INSTRUMENTATION_ENABLED=host 付きで Agent インストール → アプリ再起動

terraform output demo_step1_before 〜 demo_step4_after で各ステップのコピペ用コマンドを出力します。

期待されるトレース（After フェーズ）

  • Service Map: pov-java-front → pov-py-back → postgres の 3 階層
  • /db/normal: 1 トレース内に postgres.query span が 1 個
  • /db/n1: 1 トレース内に postgres.query span が 21 個（N+1 アンチパターン）
  • /db/long-run: postgres.query span が 35 秒幅で表示
  • /analytics: 正常時は Java → Python → postgres の 2 DB spans。エラー時は Java span 内に Python 呼び出しが 2 回（500ms 待機リトライ）観測される
  • /analytics/summary: cohort=0（1/7 の確率）で全コホートが ZeroDivisionError → 全スパンがエラー
  • Logs: dd.trace_id が埋まり、トレース ↔ ログ相関が動く

Operations

Apply / Destroy

cd terraform
terraform init
terraform plan
terraform apply     # RDS 作成に 5〜10 分
terraform destroy   # 残骸なしで完全削除

Destroy 容易化: RDS は skip_final_snapshot / deletion_protection=false / backup_retention_period=0 / delete_automated_backups=true / Performance Insights・Enhanced Monitoring・CloudWatch Logs export を全無効化。Secrets Manager は不使用（回復待機期間問題を回避）。

EC2 アクセス

ssh -i terraform/keys/pov-ssh-key ubuntu@<java_public_ip>
ssh -i terraform/keys/pov-ssh-key -J ubuntu@<java_public_ip> ubuntu@<python_private_ip>
